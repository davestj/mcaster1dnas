macOS Build Guide

Mcaster1DNAS — macOS ARM64 Build

Complete instructions for compiling the server binary (Autotools) and native Qt6 GUI application on Apple Silicon (arm64) and Intel Macs using Homebrew.

◆ Version 2.5.3-beta ◆ Updated: 2026-03-06 ◆ Branch: macos-dev ◆ Platform: Darwin arm64 (Apple M-series) ◆ Server: Autotools • GUI: CMake + Qt6

⚡ Quick Start (Apple Silicon)

# Install all dependencies (server + Qt6 GUI)
brew install autoconf automake libtool pkg-config qt \
  libogg libvorbis theora speex openssl@3 \
  libxml2 libxslt libyaml curl

# Build the server binary
./build-macos.sh          # detects macOS, sets all Homebrew paths
./build/bin/mcaster1 -c mcaster1dnas-console.yaml

# Build + launch the Qt6 GUI application
cmake -B macos/build-qt -S macos -DCMAKE_PREFIX_PATH=$(brew --prefix qt) -DCMAKE_BUILD_TYPE=Release
cmake --build macos/build-qt -j$(sysctl -n hw.logicalcpu)
codesign --force --sign - macos/build-qt/Mcaster1DNAS.app
open macos/build-qt/Mcaster1DNAS.app

Project Overview

Mcaster1DNAS (Digital Network Audio Server) is a fork of Icecast-KH / Icecast 2.x, extended with ICY 2.2 extended metadata, YAML configuration, per-listener SSL enforcement, static mount point definitions, and a REST song-history JSON API.

The primary server is written in C (cross-platform), targeting Linux, macOS, and Windows. This guide covers macOS builds using GNU Autotools.

Platform Status

PlatformArchStatusBuild Tool
macOS (Apple Silicon) arm64 Active Autotools + Homebrew
macOS (Intel) x86_64 Supported Autotools + Homebrew
Linux (Debian/Ubuntu) x86_64 / arm64 Active Autotools + apt
Linux (RHEL/Fedora) x86_64 Supported Autotools + dnf
Windows (VS 2022) x64 Active MSBuild + vcpkg
Note on arm64: This guide targets Apple Silicon (M1, M2, M3, M4 chips). Homebrew installs to /opt/homebrew on arm64 vs /usr/local on Intel. The build-macos.sh script detects the active Homebrew prefix automatically via brew --prefix, so the same script works on both architectures.

Why Plain ./configure Fails on macOS

Running ./configure without arguments fails on macOS Apple Silicon with: 

configure: error: must have Ogg Vorbis v1.0 or above installed

The root cause is that the legacy Xiph autoconf macros (ogg.m4, vorbis.m4, theora.m4, speex.m4) hard-code /usr/local/include as their fallback header search path and do not call pkg-config. On Apple Silicon, Homebrew installs to /opt/homebrew, so the macros never find ogg/ogg.h.

Affected macros and their fix

MacroDefault fallbackFix (passed to configure)
m4/ogg.m4 /usr/local --with-ogg=$(brew --prefix)
m4/vorbis.m4 /usr/local --with-vorbis=$(brew --prefix)
m4/theora.m4 /usr/local --with-theora=$(brew --prefix)
m4/speex.m4 /usr/local --with-speex=$(brew --prefix)
m4/xiph_openssl.m4 calls pkg-config Works; add keg-only path to PKG_CONFIG_PATH

Additionally, libxml2, openssl@3, and curl are keg-only formulae (Homebrew does not link them into its main prefix because macOS ships system versions). We must add their individual pkgconfig directories to PKG_CONFIG_PATH.

Step 2 — Build with build-macos.sh

Step 2 of 5

The recommended build path uses build-macos.sh, which detects the OS, verifies Homebrew packages, sets all required environment variables, and runs configure + make automatically.

Basic build (output to ./build/)

./build-macos.sh

Common options

FlagEffect
--prefix /usr/localInstall to /usr/local instead of ./build/
--cleanRun make clean before building
--no-installCompile only; skip make install
--regenerateForce ./autogen.sh before configure

Examples

./build-macos.sh --regenerate --clean

Step 3 — Manual Configure (Advanced)

Step 3 of 5

If you prefer to run configure directly, we must export the Homebrew paths before calling it. The environment variables below are what build-macos.sh sets internally.

Apple Silicon (arm64 — Homebrew at /opt/homebrew)

# Capture the active Homebrew prefix once
export BREW="$(brew --prefix)"

# Keg-only packages need explicit pkgconfig paths
export PKG_CONFIG_PATH="${BREW}/lib/pkgconfig:${BREW}/opt/openssl@3/lib/pkgconfig:${BREW}/opt/curl/lib/pkgconfig:${BREW}/opt/libxml2/lib/pkgconfig:${BREW}/opt/libxslt/lib/pkgconfig:${BREW}/opt/libyaml/lib/pkgconfig"

export CPPFLAGS="-I${BREW}/include -I${BREW}/opt/openssl@3/include -I${BREW}/opt/libxml2/include/libxml2"
export LDFLAGS="-L${BREW}/lib -L${BREW}/opt/openssl@3/lib -L${BREW}/opt/libxml2/lib"

# Run configure; --with-ogg/vorbis/theora/speex point the legacy Xiph m4 macros
# to the Homebrew prefix instead of their /usr/local fallback
./configure \
    --prefix="${HOME}/mcaster1-dev" \
    --with-ogg="${BREW}" \
    --with-vorbis="${BREW}" \
    --with-theora="${BREW}" \
    --with-speex="${BREW}" \
    --with-yaml

make -j$(sysctl -n hw.logicalcpu)

Intel Mac (x86_64 — Homebrew at /usr/local)

# On Intel, Homebrew matches the legacy /usr/local fallback, so --with-XXX
# flags are optional. keg-only PKG_CONFIG_PATH still required.
export BREW="$(brew --prefix)"    # resolves to /usr/local on Intel

export PKG_CONFIG_PATH="${BREW}/opt/openssl@3/lib/pkgconfig:${BREW}/opt/curl/lib/pkgconfig:${BREW}/opt/libxml2/lib/pkgconfig"
export CPPFLAGS="-I${BREW}/opt/openssl@3/include"
export LDFLAGS="-L${BREW}/opt/openssl@3/lib"

./configure --prefix="${HOME}/mcaster1-dev" --with-yaml
make -j$(sysctl -n hw.logicalcpu)

Linux (Debian/Ubuntu)

# Install system packages first
sudo apt-get install -y \
    autoconf automake libtool pkg-config \
    libogg-dev libvorbis-dev libtheora-dev libspeex-dev \
    libssl-dev libxml2-dev libxslt1-dev libyaml-dev libcurl4-openssl-dev

# On Linux, pkg-config finds everything; no env var overrides needed
./configure --prefix=/usr/local --with-yaml
make -j$(nproc)
sudo make install

Linux (RHEL/Fedora)

sudo dnf install -y \
    autoconf automake libtool pkgconf-pkg-config \
    libogg-devel libvorbis-devel libtheora-devel speex-devel \
    openssl-devel libxml2-devel libxslt-devel libyaml-devel libcurl-devel

./configure --prefix=/usr/local --with-yaml
make -j$(nproc)
sudo make install

Platform Differences — Source Code Compatibility

The server source is already written for cross-platform compatibility. The table below documents the Linux-specific APIs used in the codebase and how each one is handled on macOS.

Feature Linux API macOS handling Guard
Signal I/O signalfd() Excluded at compile time; traditional sigaction() used instead #ifdef HAVE_SIGNALFD
Close-on-exec pipe pipe2(fd, O_CLOEXEC) Falls back to pipe() + fcntl(FD_CLOEXEC) #ifdef HAVE_PIPE2
Privilege drop setresuid/setresgid Falls back to POSIX setuid/setgid #ifdef HAVE_SETRESUID
Executable path /proc/self/exe _NSGetExecutablePath() from mach-o/dyld.h #ifdef __APPLE__
Epoll / I/O events epoll_* Not used; codebase uses POSIX poll() throughout N/A (poll is portable)
High-res clock clock_gettime() Available on macOS 10.12+; detected by configure HAVE_CLOCK_GETTIME
Good news: We do not use epoll anywhere in the server. The connection loop uses POSIX poll(), which works identically on Linux, macOS, and all BSD derivatives. No kqueue shim is needed.

Step 4 — Regenerating Build Files (autogen.sh)

Step 4 of 5

macOS note: Homebrew installs GNU libtool as glibtoolize because macOS ships its own libtoolize stub. The updated autogen.sh detects Darwin and selects glibtoolize automatically.

Regenerate the build system

Troubleshooting

configure fails: "must have Ogg Vorbis installed"

We did not pass --with-ogg / --with-vorbis to configure, or the wrong Homebrew prefix was used. Use build-macos.sh which handles this automatically.

configure: error: must have Ogg Vorbis v1.0 or above installed

# Verify libogg is installed
brew list libogg
pkg-config --cflags --libs ogg

# Fix: use build-macos.sh
./build-macos.sh --clean --regenerate

autogen.sh fails: "libtoolize: command not found"

Homebrew libtool provides glibtoolize, not libtoolize. The updated autogen.sh auto-detects this on Darwin. If needed, override manually:

brew install libtool
LIBTOOLIZE="$(brew --prefix)/opt/libtool/bin/glibtoolize" ./autogen.sh

SSL / OpenSSL errors at configure

openssl@3 is keg-only. The PKG_CONFIG_PATH must include its pkgconfig directory explicitly:

export PKG_CONFIG_PATH="$(brew --prefix)/opt/openssl@3/lib/pkgconfig"
./configure ...

Linker errors: "library not found for -lssl"

export LDFLAGS="-L$(brew --prefix)/opt/openssl@3/lib"
./configure ...

Check what configure detected

# Read the generated config.h summary
grep -E "HAVE_OPENSSL|HAVE_YAML|HAVE_OGG|HAVE_VORBIS|HAVE_THEORA" config.h

Check full configure output

tail -100 config.log | grep -E "error|warning"

Key Architecture Notes

  • In-process server — ServerThread calls server_init() + server_process() on a QThread; shares global.* with UI.
  • Config discovery — walks 6 directory levels up from .app/Contents/MacOS/ to find mcaster1dnas.yaml at repo root.
  • No config_clear() before server starts — that function calls global_lock() which aborts if server_init() has never run. ConfigDialog::loadFromFile() uses config_parse_file() on a local mc_config_t without locking.
  • Safe config_parse_file() from GUI thread — config_parse_file() calls log_write() which needs the logging rwlock alive. Pattern: init_log_subsys() → config_parse_file(...) → log_shutdown(). Both ConfigDialog::loadFromFile() and discoverServerUrl() use this pattern. Required includes in extern "C" block: logging.h, log/log.h.
  • OverviewHeader banner — header-banner-v2.png (1926×142 px) is pre-scaled to half height at load (scaledToHeight(h/2) = 71 px) and drawn to fill widget width via p.drawPixmap(0,0,W,H,m_px). The status badge (bW=50, bH=61) is sized proportionally to the 71 px banner height. These two numbers are coupled — always scale together.
  • Codesign after every rebuild — Apple Silicon caches the code-directory. After cmake --build replaces the binary, run codesign --force --sign - macos/build-qt/Mcaster1DNAS.app or the app receives SIGKILL on launch.
  • Toolbar — ⚙ Config | ▶ Start | ■ Stop | 🌐 Web UI | 🔧 Admin | [status icon] | [uptime]
  • QtWebEngineWidgets — Credits and Help tabs use QWebEngineView. Docs are copied to Contents/Resources/docs/ (not MacOS/ — codesign rejects non-code files there) via CMake POST_BUILD step.

Critical Bug Fixes — macOS (macos-dev branch)

1. src/xslt.c — XSLT crash on libxml2 ≥ 2.12 Fixed 2.5.3-beta

Symptom: Server worker thread crashes with _platform_memmove → xmlBufAdd → xmlOutputBufferWrite → htmlDocContentDumpFormatOutput → xsltSaveResultTo whenever a browser or curl fetches any XSLT page (e.g. /status.xsl).

Root cause: The old xslt_SaveResultToBuf() created a custom output buffer with xmlOutputBufferCreateIO(encoder=NULL) then passed it to xsltSaveResultTo(). On libxml2 ≥ 2.12 (Homebrew 2.15.1), the internal xmlBuf allocation strategy changed: htmlDocContentDumpFormatOutput calls xmlBufAdd on a buffer whose memory state is incompatible with the new IO-buffer initialization, crashing in _platform_memmove.

Fix: xslt_SaveResultToBuf() now uses xsltSaveResultToString() (guarded by #ifdef HAVE_XSLTSAVERESULTTOSTRING, always defined when Homebrew's libxslt ≥ 1.1.18 is present) and copies the result into a single refbuf_t. The old xmlOutputBufferCreateIO path is retained in the #else branch for systems with old libxslt.

2. src/cfgfile.h + cfgfile.c — const xmlError* Fixed 2.5.2-dev

libxml2 2.12+ changed xmlStructuredErrorFunc to pass a const xmlError *. Fixed by changing the parameter type in config_xml_parse_failure().

3. src/slave.c — CURLOPT_XFERINFOFUNCTION return type Fixed 2.5.2-dev

icecurl_server_running() returned size_t; curl expects int. Fixed return type.

4. src/xslt.c — double-free after xsltSaveResultTo on libxml2 ≥ 2.12 Fixed 2.5.2-dev

htmlSaveFileTo now closes the output buffer internally in libxml2 ≥ 2.12. Added #if LIBXML_VERSION < 21200 guard around xmlOutputBufferClose(buf). (Superseded by fix #1 above which avoids the custom IO buffer entirely.)

5. autogen.sh — glibtoolize detection Fixed 2.5.2-dev

Homebrew installs libtoolize as glibtoolize. autogen.sh now probes both names and falls back gracefully.

6. Qt6 GUI — Config button crash (NULL logging rwlock) Fixed 2.5.3-beta

Symptom: Clicking the ⚙ Config toolbar button crashes with KERN_INVALID_ADDRESS at 0x0 when the server has not yet been started.

Root cause: config_parse_file() calls log_write() internally. log_write() acquires _logger_rwl (a pthread rwlock), which is NULL until init_log_subsys() runs — normally called inside initialize_subsystems() when the server starts. Calling config_parse_file() from the GUI thread before the server starts dereferences a NULL rwlock.

Fix: Wrap every config_parse_file() call in GUI code with init_log_subsys() before and log_shutdown() after. Add #include "logging.h" and #include "log/log.h" to the extern "C" block. Applied to ConfigDialog::loadFromFile() and MainWindow::discoverServerUrl().

7. src/log/log.c — log_close() latent NULL mutex crash Fixed 2.5.3-beta

Symptom: log_shutdown() crashes with KERN_INVALID_ADDRESS at 0x10 (= offset of sys_mutex in mutex_t without THREAD_DEBUG: 8-byte char* + 4-byte int + 4-byte padding = 16 = 0x10).

Root cause: Without log_callback (no log worker thread), log_close() called _log_close_internal() on all 20 pre-allocated loglist[] entries — including unused ones whose mutex was never initialized (remains NULL pointer). The _log_close_internal() function contract requires “enter with q lock” — but log_close() never called _lock_q() on the non-callback path, so even the contract was violated.

Fix: In src/log/log.c::log_close(), non-callback path: 

// 1. Skip unused (uninitialized) entries
if (loglist [log_id].in_use == 0) break;
// 2. Satisfy _log_close_internal() contract ("enter with q lock")
_lock_q (log_id);
_log_close_internal (log_id);

8. Overview tab — header banner scaling Fixed 2.5.3-beta

Symptom: header-banner-v2.png (1926×142 px) rendered at full natural height dominated the Overview tab. Status indicator badge sized independently of the banner caused mismatched proportions.

Fix: Pre-scale the pixmap to half height at load time: m_px = m_px.scaledToHeight(m_px.height()/2, Qt::SmoothTransformation) → 71 px. Draw to fill widget width: p.drawPixmap(0, 0, W, H, m_px). Badge proportionally sized to 71 px: bW=50, bH=61, bY=(H-bH)/2. These two values are coupled — always scale together.
