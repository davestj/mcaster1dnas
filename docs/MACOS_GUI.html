macOS Qt6 GUI Guide

Mcaster1DNAS.app — native macOS ARM64 desktop application
v2.5.3-beta Qt6 6.x macOS ARM64 Apple Silicon

Overview

Mcaster1DNAS.app is the native macOS GUI for Mcaster1DNAS, built with Qt6. It runs the full streaming server in-process (same model as the Windows MFC GUI), providing a unified desktop experience without requiring a separate terminal session or daemon. Requires macOS 12 or later on Apple Silicon (M-series).

  • Full in-process Mcaster1DNAS server (no separate process or daemon)
  • Start/Stop server from toolbar with live status indicator
  • Real-time stats via StatsPoller (calls stats_get_xml() every 5 seconds)
  • Live log tailing for Access, Error, and Playlist logs with color-coded severity
  • Server console output capture via pipe (stdout/stderr)
  • 6-tab Configuration Manager dialog (Global, Mounts, Relays, System, Security, SSL Cert)
  • Config button automatically disabled while server is running (thread-safe)
  • Web UI and Admin toolbar buttons open browser to the live server
  • System tray icon with Start / Stop / Show Window / Quit menu
  • ICY 2.2 structured metadata fields (70+ keys across 8 groups) for mount points
  • Integrated Credits and Help documentation tabs

Building from Source

The macOS build uses the autotools chain for the server core and delegates to CMake internally for the Qt6 GUI. Always use the make gui target — do not invoke CMake directly.

# Step 1 -- Regenerate build system
./autogen.sh

# Step 2 -- Configure (Apple Silicon / Homebrew)
BREW=/opt/homebrew
PKG_CONFIG_PATH="$BREW/lib/pkgconfig:$BREW/opt/openssl@3/lib/pkgconfig:$BREW/opt/curl/lib/pkgconfig:$BREW/opt/libxml2/lib/pkgconfig:$BREW/opt/libxslt/lib/pkgconfig:$BREW/opt/libyaml/lib/pkgconfig" \
CPPFLAGS="-I$BREW/include -I$BREW/opt/openssl@3/include -I$BREW/opt/libxml2/include/libxml2 -I$BREW/opt/curl/include -I$BREW/opt/libyaml/include" \
LDFLAGS="-L$BREW/lib -L$BREW/opt/openssl@3/lib -L$BREW/opt/libxml2/lib -L$BREW/opt/curl/lib" \
./configure --prefix=$(pwd)/build \
  --with-ogg=$BREW --with-vorbis=$BREW --with-theora=$BREW --with-speex=$BREW \
  --with-xslt-config=$BREW/opt/libxslt/bin/xslt-config \
  --with-yaml

# Step 3 -- Build server binary
make -j$(sysctl -n hw.logicalcpu)

# Step 4 -- Build Qt6 GUI (delegates to cmake internally)
make gui

# Step 5 -- Codesign (required on Apple Silicon after every rebuild)
codesign --force --sign - macos/build-qt/Mcaster1DNAS.app

# Step 6 -- Launch
open macos/build-qt/Mcaster1DNAS.app
IMPORTANT: Always use ./autogen.sh → ./configure → make → make gui. Do not invoke cmake directly. The make gui target handles cmake configuration with the correct Homebrew PKG_CONFIG_PATH to avoid ABI mismatches between system libxslt and Homebrew libxml2.

For convenience, build-macos.sh in the repository root performs the full configure and build sequence automatically, including platform detection and Homebrew package verification.

Critical Homebrew Dependencies

Install all required packages before building. Several packages are keg-only and must be explicitly referenced via PKG_CONFIG_PATH — the configure command above and build-macos.sh handle this automatically.

The server and GUI are compiled into a single binary. There is no companion daemon or helper process — server_init() and server_process() run on a dedicated QThread (ServerThread) inside the same process as the Qt event loop.

Configuration Manager

The Configuration Manager is a 6-tab QDialog opened from the toolbar Config button or via Tools → Configuration Manager. It is disabled (grayed out) while the server is running, because modifying config while the server holds C-level mutexes is not thread-safe.

When the server is running, the Config button is grayed out. Stop the server first to open the Configuration Manager, make changes, save, then restart.

Dialog Tabs

Tab Contents
Global Hostname, admin email, server-id, fileserve toggle, listen sockets table (add/remove port + bind address + SSL entries), source and relay authentication passwords
Mounts Add / Edit / Duplicate / Remove mount points. Each mount opens a 5-tab editor: Basic, Stream Info, Access Control, Advanced, and ICY2 Metadata (70+ keys across 8 groups)
Relays Relay server management: remote server, port, mount, SSL toggle, on-demand relay, ICY meta relay, check interval
System Resource limits, connection timeouts, queue and burst sizes, file paths, logging configuration (log level, rotate size)
Security Process isolation (chroot path, run-as user/group), TLS (certificate path, key path, CA bundle, cipher list), HTTP response headers table
SSL Cert Inspect the current PEM certificate (subject, issuer, validity dates, SANs via OpenSSL X.509 API) and generate a new self-signed certificate or CSR via ssl_gen_run()

Save Behavior

  • Save — writes to the active config file in the auto-detected format (YAML or XML, based on file extension and content)
  • Save as YAML… — writes to a new .yaml file via QFileDialog, then prompts to activate it as the running config
  • Save as XML… — writes to a new .xml file via QFileDialog, then prompts to activate it
  • After saving (server stopped): dialog confirms the config is saved and ready for the next Start

Config Auto-Discovery

When launched from Finder, the Dock, or a shortcut, macOS sets the working directory to $HOME. The app uses a discovery algorithm to locate the config file regardless of how it was launched:

  1. -c <path> command-line argument (if provided)
  2. <binary dir>/mcaster1dnas.yaml — directory containing the Mcaster1DNAS binary inside the .app bundle
  3. Walk up 6 parent directories from the binary path, checking each level for mcaster1dnas.yaml
  4. QFileDialog prompt — asks the user to locate the config file if none is found automatically

This handles both Finder/Dock launch (CWD=$HOME) and terminal launch (open macos/build-qt/Mcaster1DNAS.app from the repo root).

Port Defaults

Port Protocol Bind Address Config Key
9330 HTTP 127.0.0.1 listen-sockets[0].port
9443 HTTPS / TLS 127.0.0.1 listen-sockets[1].port + ssl: 1

The default config file is mcaster1dnas.yaml in the repository root. The SSL listener uses ssl/localhost.pem (combined PEM: private key block followed by certificate block). These ports are bound to 127.0.0.1 only and are not exposed on external interfaces.

Known Issues / Gotchas

Codesign after every rebuild

Apple Silicon caches the code-directory. After cmake --build replaces the binary, the cached signature is stale and macOS will SIGKILL the app on launch. Always run:

codesign --force --sign - macos/build-qt/Mcaster1DNAS.app

The make gui target performs this automatically after every build.

Do NOT use the -b (daemon) flag

fork() after multi-threaded initialization is unsafe on Darwin. The GUI manages the server lifecycle internally via ServerThread. Do not pass -b to the binary or attempt to daemonize it. Use the GUI Start button or launch as a foreground process from the terminal.

ABI mismatch: system vs Homebrew libxslt

macOS ships /usr/bin/xslt-config which links libxslt against the system libxml2 (~2.9). Homebrew's libxml2 is 2.15+ and changed internal struct layouts (xmlBuf) between these versions. Linking against the system libxslt while using Homebrew libxml2 causes a SIGBUS (ABI mismatch) whenever the server transforms an XSLT page. The make gui target and the configure command above both force Homebrew's xslt-config via --with-xslt-config.

config_clear() abort

config_clear() calls global_lock() which calls thread_mutex_lock_c(). If the server has never been initialized, the underlying pthread mutex is uninitialized and this results in abort(). Never call config_clear() on a local mc_config_t in GUI code. ConfigDialog::loadFromFile() calls config_parse_file() directly and deliberately omits config_clear().

Relative paths in YAML config

YAML config paths like ./ssl/localhost.pem, ./logs, and ./web are resolved relative to the working directory. ServerThread performs a chdir() to the directory containing the config file before calling server_init(), mirroring the SetCurrentDirectoryA(exedir) call in the Windows Mcaster1Win.cpp InitInstance(). This ensures all relative paths resolve correctly regardless of how the .app was launched.
